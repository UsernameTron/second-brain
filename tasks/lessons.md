### Learned Rules
- [2026-04-23] [Testing]: Never spawn multiple test/coverage processes in parallel. One invocation at a time, always with `--forceExit`. If a test command is backgrounded by the user, wait for completion notification — do not retry. Triggered by: 5 duplicate jest processes hung simultaneously, had to be killed manually, wasted 30+ minutes.
- [2026-04-23] [Review]: 3-reviewer cross-audit catches findings single-reviewer misses. Gemini caught dead-letter queue gap in memory promotion; Opus caught config-crash paths in loadPipelineConfig. Always request independent review from a second model when shipping milestone-boundary work.
- [2026-04-23] [Hygiene]: Config overlay pattern (pipeline.local.json) solves dev-workflow reversion. Ship hygiene changes ahead of feature milestones when they unblock local development — don't gate infrastructure fixes behind feature scope.
- [2026-04-23] [Hooks]: Stop hook excluded tasks/todo.md from clean-tree check, which allowed session-48 handoff state to survive uncommitted into the next session. Pattern works for rolling ledgers but surfaces ambiguity at milestone close. Future: have the stop hook prompt for commit at session end when milestone-boundary files are dirty.
- [2026-04-23] [Testing]: Do not run the full test suite as a gate on markdown-only commits (planning files, lessons, todo). Tests gate source code changes. Running Jest on a 4-file planning commit is ceremony, not discipline — and when tests are slow, it blocks the user repeatedly for zero value. Check the diff scope first; skip tests when no source/config files changed.
- [2026-04-24] [Verification]: Plan-text verification (gsd-verifier on PLAN.md) validates what the plan *promises*, not what the implementation actually ships. When a plan's must_haves include live-system behavior ("direct push rejected", "CI enforced"), the Phase-level verifier must run commands against the deployed system — not just grep plan files. Triggered by: Task 1 in Plan 17-03 shipped a `gh api` payload that omitted `required_pull_request_reviews`, but plan verification passed because the plan text was correct. Only Pete's post-merge UI audit caught the gap.
- [2026-04-24] [GitHub]: Classic branch protection's `required_status_checks` only gates PR merges — it does NOT block direct pushes. To force the PR path, `required_pull_request_reviews` must be non-null (set `required_approving_review_count: 0` for solo devs). Without the PR requirement, direct push with red CI still lands. Triggered by: BRANCH-PROT-01 shipped without the PR-required bit; direct pushes were still allowed until the fix.
- [2026-04-24] [Git workflow]: GitHub web UI edits bypass all local discipline and can introduce issues that fly past CI and review. Two direct-to-master commits via web UI during this session violated P11 (secret at workflow-level in ci.yml) and disabled CI triggers entirely. Prevention: keep `required_pull_request_reviews` on master (force PR path for UI edits too), review every commit to master in the PR it arrives via. Triggered by: b33ca14 and e4c080c on 2026-04-24, both reverted via PR #32.
- [2026-04-24] [Cache invalidation]: Before recommending a cache-bust / full-re-embed trigger, ask whether the changed value actually affects the cached artifact or only the downstream consumption. Embeddings are frozen vectors — threshold and scoring-decay factors are applied at query time to cosine results, so changing them does NOT require new embeddings. Defensive "hash everything into the version" logic creates unnecessary compute. Triggered by: Phase 19 discuss-phase — recommended composite schema_version (model + dim + decay + threshold), user corrected that threshold/decay are query-time math and re-embed on those is waste. Final lock: `schema_version = hash(model || dimension)` only.
- [2026-04-24] [Discuss-phase]: Don't lock decisions based on offhand operator comments before checking REQUIREMENTS.md / ROADMAP.md / prior phase context. Treat passing remarks as hypotheses to discuss, not as documented overrides. Why: operator-stated preferences from a different chat context are sometimes "casual reads" rather than reviewed architectural calls — locking them silently bypasses the documented spec. How to apply: when an operator preference contradicts a written REQ, surface the conflict before treating either as locked. Triggered by: Phase 20 file-path discussion — locked `RIGHT/metrics/daily-stats.md` from a casual operator read without checking that REQUIREMENTS.md said `RIGHT/daily-stats.md`. Operator caught it and asked to unlock for proper discussion.
- [2026-04-25] [Verification] LESSON-RECIPE-VERIFY-01: Recipes that prescribe additive changes ("add X to file Y") must include a Step 0 that reads the target file and confirms X isn't already present. Why: twice in one session (v1.4 Phase 21 pre-flight), proposed fixes were no-ops because the change already existed — Scenario-6 hooks already at lines 372–381 of semantic-search.test.js, and describeFn skip guard already at lines 149–151 of uat-classification.test.js. How to apply: any Edit/Write step in a recipe must be preceded by a Read step that proves the proposed change isn't redundant.
- [2026-04-25] [Verification] LESSON-TRIP-WIRE-RELATIVE-01: Recipe verification steps that compare test counts must specify pre-fix baseline counts explicitly OR use relative conditions ("any test that was passing before is now failing") instead of absolute ones ("0 failed total"). Why: pre-flight CHECK 2 fired on UAT-01 (a pre-existing local-only failure) during the dotenv-fix verification; the fix itself was sound but the over-strict trip-wire forced a STOP-and-stash detour. How to apply: trip-wires baseline against the most recent known state, not against a Platonic 0-failure ideal.
- [2026-04-25] [Git workflow] LESSON-SQUASH-CONTENT-CHECK-01: After a PR is squash-merged, `git log origin/master..branch` shows the original branch commits as "unique to branch" because squash creates a new SHA with no ancestry link. Use `git cherry-pick --no-commit <orig-sha>` then `git status --porcelain` to verify the diff is empty — this proves content-equivalence regardless of SHA. Why: phantom-branch recovery during this session hit a false-positive STOP because SHA comparison alone didn't see the squash content. How to apply: when verifying whether a branch's content is in master before deletion, use cherry-pick dry-run; bake this verification into recovery recipes.
- [2026-04-25] [Testing] LESSON-PREFLIGHT-CI-MODE-01: Second Brain pre-flight test checks must use `CI=true npm test`, not bare `npm test`. UAT tests use `describeFn = (!skipInCI && HAS_API_KEY) ? describe : describe.skip`; without `CI=true`, local runs hit live API calls and results depend on `.env` key validity. Why: pre-flight CHECK 2 conflated developer-environment behavior with production correctness by requiring local `npm test` to return 0 failures, forcing multiple unnecessary detours. How to apply: green/red signal for this project is `CI=true npm test`; local-mode UAT runs are developer-optional debugging, not pre-flight gates.
- [2026-04-25] [Testing] LESSON-UAT-CORPUS-DRIFT-01: Any classifier-side change (prompt, threshold, excluded terms, content-policy) must include a UAT-corpus-revalidation step. UAT corpora encode classifier-behavior contracts at the time they're written; CI guards mask drift because UAT tests skip in CI. Why: UAT-01's accuracy-aggregate test failed locally because the live classifier returned BLOCKED on cases the corpus expected to receive LEFT/RIGHT — most likely cause: excluded-terms expansion in v1.1 (15 terms, substring matching) shifted classifier decision boundaries without a corresponding corpus rebaseline. How to apply: classifier-side PRs going forward must run `unset CI; npx jest test/uat/uat-classification.test.js` and refresh fixtures if the score drops; v1.5 phase UAT-CORPUS-REFRESH-01 (see tasks/todo.md) will rebaseline the existing corpus.
- [2026-04-25] [Process] LESSON-MANIFEST-FIRST-VALIDATED-01: For any scoped governance work (categorizing N sites, applying labels to N items), produce the categorization manifest first, validate it, then apply. Drift between planning estimates and live code is the default state, not the exception. Why: D-LOCK-2-AMEND-A's manifest-first protocol on Plan 21-03 caught 5 categorization corrections from CONTEXT estimates (~16% of the 32-site corpus). Most consequential: 3 sites in memory-proposals.js (lines 319/329/336) were about to be tagged as last-resort-error but turned out to be always-emit JSON audit logs — diagnostic category, same shape as vault-gateway:159. Without the manifest gate, those 5 sites would have been wrongly tagged. How to apply: any plan whose scope is "apply label X to N items" gets a Task 0 that builds the full N-row manifest and pauses for review before any application step. Triggered by: Plan 21-03 (HYG-CONSOLE-01) execution — manifest-first protocol prevented 5 categorization errors from shipping.
- [2026-04-25] [Hygiene] LESSON-OPTION-A-SCOPE-CORRECTION-01: When an audit gate fails on pre-existing artifacts in the same Lock-class as primary work, the cleanest fix is scope correction (Option A — bring artifacts into compliance) rather than scope limitation (Option B — exempt them). Pattern 9 spirit is uniformity; partial coverage creates a worse audit surface than uniform coverage. Why: Plan 21-03's A3 audit grep flagged 3 pre-existing bare `eslint-disable-next-line no-console` directives outside the primary 9-file scope (in promote-memories.js and semantic-index.js). Bringing them into scope as a "corollary" cost ~30 seconds and brought src/ to fully uniform Pattern 9 governance state; the alternative (scope-limit A3) would have left 3 grandfathered disables that fail any future global audit. How to apply: if a discovered out-of-scope artifact is (a) same work-class as in-scope items and (b) Lock-fence permits, bring it into scope as a corollary with explicit framing in SUMMARY (e.g., "32 primary + 3 corollary"). Reserve Option B (scope-limit) or Option C (defer) for when work-class genuinely differs. Triggered by: Plan 21-03 Part 2 — Connor selected Option A; closeout PR shipped 35 uniformly category-tagged disables instead of 32 + 3 grandfathered.
- [2026-04-25] [Hygiene] LESSON-BONUS-REQ-FLIP-01: D-LOCK-4's "verifiable v1.4-fact error" rule legitimately permits adjacent-scope corrections beyond the named target rows. When a fact-error rule is the gate, apply it broadly within the file; document scope expansions explicitly in SUMMARY so future audits can trace why bonus flips landed in a non-bonus PR. Why: Plan 21-04 was scoped to flip 4 Phase 21 REQ rows; execution caught and flipped 4 Phase 20 rows (STATS-LATENCY-01, STATS-GROWTH-01, +2) that were stale Pending despite Phase 20 shipping via PR #35. Both groups satisfy "verifiable v1.4-fact error" — the rule is broader than the initial framing suggested, and leaving Phase 20 stale would have rolled forward as drift into v1.5. How to apply: when scoped to a "fact-error correction" within a single file, scan the whole file for the same class of error and fix all matches; document the scope expansion in SUMMARY §3 (or equivalent) for reviewer visibility. Triggered by: Plan 21-04 Task 4 — bonus 4 Phase 20 REQ flips landed in the v1.4 close PR alongside the 4 Phase 21 in-scope flips.
- [2026-04-25] [Process] LESSON-LIVE-RECOUNT-AT-EXECUTE-01: Any plan that updates numerical doc figures (test counts, coverage, warning counts, dependency counts) should make "re-count from live state" its first task. Downstream tasks reference the captured value, never stale estimates from CONTEXT or earlier session output. Why: numerical figures in plan briefs go stale through 3 shifts: CONTEXT-time → plan-write-time → execution-time → ship-time. Plan 21-04's Task 1 design (re-count at execution start, reference downstream) prevented all 3 shifts from creating phantom drift. Test count shifted across the session from 1082 (CONTEXT estimate) → 1066 (Plan 21-02 local check) → 1127 (Plan 21-04 execution); only the last is correct for v1.4 ship. How to apply: Task 1 of any doc-refresh plan runs the live commands (`CI=true npm test`, `find test/ -name "*.test.js" | wc -l`, `npm audit`, coverage extract) and captures values for downstream tasks; subsequent tasks must NOT re-derive numbers, NOT pull from CONTEXT, and NOT use figures from prior session output. Triggered by: Plan 21-04 — Task 1 captured 1127/1044/55/coverage-block; all 5 narrative-doc edits referenced that capture and not the stale 982/944 figures sitting in CONTEXT and the plan brief.
- [2026-04-25] [Hygiene] LESSON-PRE-EXISTING-DEBT-ABSORPTION-01: Closeout phases are the natural place to absorb pre-existing technical debt that surfaces during scoped hygiene work. Treating closeout phases as "strict-scope only" misses opportunities to ship hygiene completions that cost under 30 seconds and prevent the debt from rolling forward into the next milestone. Why: Plan 21-03 absorbed 3 bare disables (corollary); Plan 21-04 absorbed 4 stale REQ rows (bonus). Both expansions stayed within Lock-fence and matched the spirit of their respective rules. Both items would otherwise have rolled into v1.5 as inherited debt. How to apply: during a closeout phase, when an audit reveals adjacent debt that is (a) same work-class as primary scope and (b) Lock-fence permits, default to absorption with explicit framing rather than deferral. Reserve deferral for work that genuinely belongs to a future milestone (e.g., HYG-UNICODE-02 deferred to v1.5 because the work-class — full Unicode-variant matching — is genuinely a v1.5 feature, not a v1.4 hygiene fix). Triggered by: Phase 21 v1.4 close — corollary disables (Plan 21-03 PR #44) and bonus REQ flips (Plan 21-04 PR #45) absorbed during scoped hygiene work.
