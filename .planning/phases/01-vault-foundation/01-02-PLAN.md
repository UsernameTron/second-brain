    **Create `src/content-policy.js` as a CommonJS module:**

    This module handles Guard 2 (content filtering) per D-06 through D-11, with sanitization per VAULT-03.

    `escapeRegex(term)` — returns `term.replace(/[.*+?^${}()|[\]\\]/g, '\\$&')` to prevent regex injection from config-supplied terms.

    `checkContent(content, excludedTerms)` async function:
    - Stage 1: Keyword scan. For each term in excludedTerms, build regex `new RegExp('\\b' + escapeRegex(term) + '\\b', 'i')`. Test against content. No match = return `{ decision: 'PASS' }` immediately (zero-latency common case per D-06).
    - Stage 2 (fires only on Stage 1 match): Call `classifyWithHaiku(content, matchedTerm, excludedTerms)`.
    - Return values:
      - No keyword match: `{ decision: 'PASS' }`
      - Keyword match + Haiku says BLOCK: `{ decision: 'BLOCK', reason: 'Excluded content detected (term: X)', matchedTerm: 'X' }`
      - Keyword match + Haiku says ALLOW: `{ decision: 'PASS' }`
      - Keyword match + Haiku error/timeout: `{ decision: 'BLOCK', reason: 'Haiku unavailable, queued for review (error message)', matchedTerm: 'X' }`

    `sanitizeContent(content, excludedTerms)` function:
    - Attempts to remove excluded references from content while preserving the rest.
    - For each excluded term found (case-insensitive, word-boundary), replace the sentence containing it with `[REDACTED]`. A "sentence" is text between sentence-ending punctuation (`.`, `!`, `?`, or newline).
    - Returns `{ sanitized: string, redactedCount: number }`.
    - This is the "strip" path that VAULT-03 requires. Used when the content filter decides to BLOCK: the gateway will attempt sanitization first, and only fully block (quarantine with no content) if sanitization removes more than 50% of the content (indicating the content is primarily about excluded topics).

    `classifyWithHaiku(content, matchedTerm, excludedTerms)` async function:
    - **Privacy-preserving context window (addresses review HIGH concern #4):** Instead of sending full content, extract a window of 100 characters before and after each keyword match. Send only these context windows to Haiku, not the full note body. Format: `"...{100 chars before}[MATCHED_TERM]{100 chars after}..."` for each match (max 3 windows).
    - Uses `@anthropic-ai/sdk`: `const Anthropic = require('@anthropic-ai/sdk'); const client = new Anthropic();`
    - Calls `client.messages.create()` with:
      - `model: 'claude-haiku-4-5'`
      - `max_tokens: 10`
      - `timeout: 2000` (2s per D-08)
      - System prompt with BLOCK/ALLOW examples per research Pitfall 3:
        ```
        You are a content filter. Classify the following content excerpt as BLOCK or ALLOW.
        BLOCK: content that is about or from these organizations — their architecture, internal processes, client data, confidential strategy, or internal people: ${excludedTerms.join(', ')}
        ALLOW: neutral tool references ("tracked in Asana"), career narrative ("led a team at Genesys"), generic industry mentions.

        BLOCK examples:
        - "The Genesys Cloud routing configuration for ISPN queue handling..." (architecture/operations)
        - "Asana project ID 12345 with client Acme..." (client data)

        ALLOW examples:
        - "I led a 12-person contact center team at Genesys." (career narrative)
        - "Tracked in Asana" (neutral tool reference)
        - "Genesys is a common CCaaS vendor in the enterprise market." (generic industry mention)

        Respond with only the word BLOCK or ALLOW.
        ```
      - User message: `"Classify these excerpts around the term '${matchedTerm}':\n\n${contextWindows.join('\n\n')}"` — never the full note body.
    - Parse response: if `response.content[0].text.trim().toUpperCase() === 'BLOCK'` return `'BLOCK'`, else return `'ALLOW'`.
    - On any error (timeout, API down, malformed response): throw the error. The caller catches and returns BLOCK.

    Exports: `module.exports = { checkContent, classifyWithHaiku, sanitizeContent, escapeRegex };`

    **Create `src/style-policy.js` as a CommonJS module:**

    This module handles Guard 3 (style lint) per D-12 through D-15 and enforces VAULT-04.

    `extractBannedWords(styleGuideContent)` function:
    - Parse the style guide markdown to find banned words/phrases. Look for a table in a section containing "Banned" in the heading (e.g., `## Banned Words` or similar).
    - Extract entries from the first column of each table row (skip header and separator rows identified by `---`).
    - Return lowercase trimmed array. Expected to include: `genuinely`, `honestly`, `straightforward`, `dive in`, `game-changer`, `leverage`, `synergy`, `innovative`, `cutting-edge`, `seamlessly`, `empower`, `unlock`, `robust`, `of course`, `certainly`, `absolutely`, etc.

    `loadStyleGuide()` function:
    - Reads `path.join(VAULT_ROOT, 'ABOUT ME', 'anti-ai-writing-style.md')` via `fs.readFileSync('utf8')`.
    - Returns `{ fullText: string, bannedWords: string[] }` where bannedWords is the result of extractBannedWords.
    - Cache result in module-level variable. Watch file with `fs.watch` and reload on change (per D-13). Use same 50ms debounce pattern as config watcher from Plan 01.

    `getStyleGuideForPrompt()` function:
    - Returns the cached `fullText` from loadStyleGuide() for system prompt injection (per D-12). Called by createVaultWriter and by downstream consumers in future phases.

    `getBannedWords()` function:
    - Returns the cached `bannedWords` array from loadStyleGuide(). Used by checkStyle.

    `checkStyle(content, bannedWords, attemptCount)` function:
    - **attemptCount contract (addresses review MEDIUM concern):** `attemptCount` is a required number parameter. The gateway does NOT track attempt state internally. The caller MUST pass `0` for first attempt, `1` (or higher) for retries. If `attemptCount` is `undefined` or not a number, throw `TypeError('attemptCount must be a number — caller must track retry state')`.
    - For each banned word, build regex `new RegExp('\\b' + escapeRegex(word) + '\\b', 'i')` and test against content. (Latency target <10ms per D-13.)
    - No match: return `{ decision: 'PASS' }`.
    - Match + attemptCount === 0: return `{ decision: 'REJECT', reason: "Banned word detected: 'X' -- regenerate without it", violation: 'X' }` (per D-14 first violation).
    - Match + attemptCount > 0: return `{ decision: 'QUARANTINE', reason: "Repeated style violation: 'X'", violation: 'X' }` (per D-14 second violation).

    `createVaultWriter(agentName)` function (addresses review HIGH concern #5 — VAULT-04 enforcement):
    - Returns an object `{ write, getSystemPromptPrefix }`:
      - `getSystemPromptPrefix()` — returns the full style guide text (from getStyleGuideForPrompt()) wrapped with instructions: `"[STYLE GUIDE — All vault content MUST follow these rules]\n\n${styleGuide}\n\n[END STYLE GUIDE]\n\nYou are writing as agent '${agentName}'. All output destined for the vault must follow the style guide above."`. This is the mechanism that enforces D-12 — every vault-writing agent calls createVaultWriter and uses the returned prefix in its system prompt.
      - `write(relativePath, content, options)` — calls through to the gateway's `vaultWrite` with the agentName recorded for audit. Same signature as vaultWrite.
    - **Enforcement mechanism:** Future phases (Phase 2+) that generate vault content MUST use `createVaultWriter('agentName')` instead of calling `vaultWrite` directly. The wrapper ensures the style guide is available. Direct `vaultWrite` calls still work (for non-generated content), but `createVaultWriter` is the required entry point for LLM-generated output.

    Import `escapeRegex` from content-policy (or duplicate the one-liner). Import `VAULT_ROOT` from vault-gateway.

    Exports: `module.exports = { checkStyle, extractBannedWords, getStyleGuideForPrompt, getBannedWords, loadStyleGuide, createVaultWriter };`
