Apply the branch protection rule via `gh api` (PUT method on the protection endpoint):

```bash
gh api \
  --method PUT \
  -H "Accept: application/vnd.github+json" \
  "repos/$REPO/branches/master/protection" \
  --input - <<'EOF'
{
  "required_status_checks": {
    "strict": true,
    "contexts": ["test (20)", "test (22)", "Analyze"]
  },
  "enforce_admins": false,
  "required_pull_request_reviews": null,
  "restrictions": null,
  "allow_force_pushes": false,
  "allow_deletions": false,
  "required_conversation_resolution": false,
  "lock_branch": false,
  "allow_fork_syncing": false
}
EOF
```

Field-by-field rationale (this matters for verification):

- `required_status_checks.strict: true` — branches must be up-to-date with master before merge (forces rebase/merge of latest master); standard GitHub "require branches to be up to date before merging" toggle.
- `required_status_checks.contexts` — the exact list of status check names. `test (20)` and `test (22)` come from the ci.yml matrix; `Analyze` comes from codeql.yml. UAT is intentionally OMITTED per BRANCH-PROT-01 ("UAT workflow is informational, not required").
- `enforce_admins: false` — Pete is the sole admin and may need emergency push access (e.g., to fix a broken master after a hook failure). The protection blocks PR merges and force-pushes for everyone but allows admin override for genuine emergencies. This matches the "solo developer + Claude" workflow philosophy in CLAUDE.md.
- `required_pull_request_reviews: null` — Pete is the only reviewer; requiring a separate reviewer would block all merges.
- `restrictions: null` — no user/team push restrictions (single-user repo).
- `allow_force_pushes: false` — REQUIRED by BRANCH-PROT-01 ("blocks force-push").
- `allow_deletions: false` — prevents accidental deletion of master.
- `required_conversation_resolution: false` — Pete reviews his own PRs; no thread-resolution gate needed.
- `lock_branch: false` — master is mutable via PR merges; only direct/force pushes are blocked.

After the API call succeeds, capture the active rule as a snapshot:

```bash
gh api "repos/$REPO/branches/master/protection" > .planning/phases/17-uat-ci-infrastructure/branch-protection-snapshot.json
```

This snapshot is the evidence artifact. It is committed to the planning directory so future audits can verify the rule was active at this point in time. The snapshot may include fields the API returned that we did not explicitly set (e.g., `url`, `nodes` arrays); that is normal — the verification only checks the fields we care about.

If the `gh api` call fails with a 403/404 (insufficient permissions), STOP and surface the error. The fix is "Pete must run `gh auth refresh -s admin:repo_hook` to grant admin scope to the local gh CLI session" and is reported back via the checkpoint task below.
